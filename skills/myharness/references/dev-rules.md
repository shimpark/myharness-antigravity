# 개발 규칙 (생성 에이전트 작업 원칙 주입용)

LLM 코딩 흔한 실수를 줄이는 행위 가이드. 생성된 빌더/수정 에이전트의 `## 작업 원칙`에 주입한다. 가치는 *생성되는 에이전트 산출물에 규칙을 전파*하는 데 있다 — 서브에이전트는 메인 세션의 글로벌 규칙을 못 받으므로 명시 주입 필요.

**트레이드오프:** 속도보다 신중. 사소한 작업엔 판단껏(과적용 금지).

## 1. 코딩 전 사고
- 가정은 명시. 불확실하면 질문.
- 해석이 여럿이면 제시 — 말없이 하나 고르지 마라.
- 더 단순한 길 있으면 말하라. 정당하면 반론.
- 불명확하면 멈추고 무엇이 헷갈리는지 말하고 질문.
- **단, 질문은 절제** — 답이 바뀔 때만 묻는다. 작고 되돌릴 수 있으면 가정을 명시하고 진행(과잉질문 금지). 질문은 데이터 손실·보안·대규모 재작업 위험이 있을 때만.

## 2. 단순성 우선
- 요청 외 기능·단일 사용처 추상화·미요청 유연성·불가능 시나리오 에러처리 전부 금지.
- 200줄을 50줄로 줄일 수 있으면 다시 써라.
- 자문: "시니어가 과설계라 할까?" → 그렇다면 단순화.

## 3. 외과적 변경
- 인접 코드/주석/포맷 "개선" 금지. 안 망가진 것 리팩토링 금지. 기존 스타일 따르라.
- 무관한 죽은 코드는 언급만, 삭제 X. 네 변경이 만든 고아만 제거.
- 테스트: 바뀐 모든 줄이 요청에 직접 추적되는가.

## 4. 목표 주도 실행
- "검증 추가"→"잘못된 입력 테스트 작성 후 통과". "버그 수정"→"재현 테스트 후 통과". "리팩토링"→"전후 테스트 통과 보장".
- 다단계는 간단 계획: `[단계] → 검증: [확인]`.

## 5. 의존성 신중
- 새 의존성(라이브러리·패키지)은 **요청받았거나, 이득이 비용보다 명백하고 이유를 설명할 때만** 추가.
- 우선 표준 라이브러리·기존 의존성으로 해결. 추가 시 버전·사유 명기.

## 6. 추측성 아키텍처 금지
- 요청이 실제로 요구하지 않으면 framework·범용 manager·adapter·factory·plugin 시스템 만들지 마라.
- 직접적이고 읽기 쉬운 코드부터. 추상화는 둘째 사용처가 생길 때.

## 주입 방법 (F1 — 실경로)
팩토리는 이 파일을 타겟 하네스의 `.claude/skills/{harness-name}/references/dev-rules.md`로 **복사**한 뒤, 에이전트 정의 `## 작업 원칙`에 **타겟 실경로** 한 줄을 넣는다: `> 개발 규칙: \`.claude/skills/{harness-name}/references/dev-rules.md\` 준수.` 서브에이전트는 `[[ ]]`·플러그인 내부 경로를 해소 못 하므로 타겟상대 실경로 필수. 본문 복붙 금지(DRY). 비코드 에이전트(문서·리서치)는 dev-rules만 선택 적용(TDD 제외).
