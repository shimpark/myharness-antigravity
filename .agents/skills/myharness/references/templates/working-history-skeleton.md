# 결과서 골격 (working-history skeleton — T1)

> 생성된 하네스의 **결과서(working_history) 골격**. 언어·스택 무관. 팩토리가 타겟 `docs/{project}/working_template/working_history_template.md`로 복사하거나, 단계별 결과서를 이 구조로 쓴다.
> 파일 위치: `docs/{project}/working_history/{단계}_{한글명}_{YYYYMMDD_HHMMSS}.md` (커밋 대상 — 영속 감사 원장). 덮어쓰기 금지.
> **자기완결 불변식:** `_workspace/`(휘발)가 사라져도 이 결과서만으로 판정·근거·다음 단계가 재구성돼야 한다. 그래서 핵심 증거는 요약+`<details>` 원문으로 **이 파일에 embed**한다.

---

# {단계ID} 작업결과서: {단계 제목}

## 1. 작업 요약
- 목표 / 범위 / 완료 항목 (불릿)

## 2. 변경 파일
- 소스 · 테스트 · 설정 (경로 + 한 줄 사유)

## 3. 검증 결과
- 테스트: {통과/전체} · 회귀: {결과} · 린트: {경고 0 여부}
- (코드 도메인) RED→GREEN 매트릭스 *(해당 시)*

## 4. 미해결 / 후속
- 알려진 한계 · 백로그 (없으면 "없음")

## 5. 외부 리뷰 반영 *(외부 리뷰가 있었을 시에만 — 도구 없으면 이 섹션 생략, "내부 QA만" 1줄)*
- 판정표 digest(확인/부분/이월/기각 + 근거) · 게이트 수치 · 출처(리뷰어)
- raw 판정·프롬프트는 `_workspace/reviews/`에 링크(경로+hash). 본문 복붙 금지.
- <details><summary>리뷰어 핵심 코멘트 원문</summary> … 결정적 코멘트만 embed … </details>

## 6. 실패 컨텍스트 *(치명 실패·디버그 근거 있을 시)*
- <details><summary>핵심 실패 로그 / stack trace</summary> … 결정적 스니펫만 … </details>

## 7. 다음 단계 참조
> ⚠️ heading 문자열 `## 다음 단계 참조` **그대로 유지** — RAG 연속성 진입점(SKILL 5-1). 다음 단계 사전작업이 `docs/{project}/working_history/`의 **최신 결과서**에서 이 블록을 먼저 읽는다.
- 미해결 이슈 · 핵심 결정과 **이유** · 다음 단계 착수 조건·주의

## 작성 규칙 (slim)
- **자동 생성 우선** — 테스트·변경파일·리뷰 digest는 자동 수집. 수동 입력 최소(단계당 ~300토큰).
- **조건부 섹션은 해당 없으면 생략** — 5·6은 마커대로. 빈 섹션 억지 작성 금지(특히 외부 도구 없는 사용자).
- **T1은 이 1장으로 끝.** main_todolist·단계별 plan은 T2(단계형/영속화 요청) 때만 — 현재 T2는 설계 승인·미구현(`factory-map.md`).
